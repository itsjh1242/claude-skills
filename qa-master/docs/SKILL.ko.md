# QA Master

## 설치 (Setup)

### Step 0 — 버전 확인 (필수 — 건너뛰지 않는다)
다른 작업 전에 반드시 아래 단계를 모두 수행한다:
1. 현재 프로젝트의 `.claude/skills/qa-master/SKILL.md` 파일을 읽는다
2. 파일이 존재하지 않으면 → 신규 설치. Step 1로 이동.
3. 파일이 존재하면 → YAML frontmatter에서 `version:` 값을 추출
   - `version` 필드가 없으면 → `0.0.0`으로 간주
4. 로컬 버전과 `2.0.0` (이 파일의 버전)을 비교:
   - 로컬 < `2.0.0` → **업데이트 흐름** 실행
   - 로컬 = `2.0.0` → "qa-master는 이미 최신 버전입니다 (v2.0.0)" 출력 후 중단. 다른 작업 하지 않는다.
5. 비교 결과를 반드시 출력한다: "로컬: v{local} / 원격: v2.0.0 → {action}"

### 업데이트 흐름
이전 버전에서 업데이트할 때:
1. `.claude/skills/qa-master/SKILL.md`를 이 파일로 덮어쓰기
2. `CLAUDE.md`의 `# QA Master` 섹션을 아래 Step 3의 최신 내용으로 교체
3. 이전 버전과 현재 버전 사이의 changelog 항목을 사용자에게 표시
4. `.dev/qa/` 디렉토리와 기존 리포트는 보존 — 삭제하지 않음
5. "qa-master 업데이트 완료: v{old} → v{new}" 출력

### Step 1 — 디렉토리/파일 생성
다음 경로를 프로젝트 루트에 생성:
- `.dev/qa/reports/`       (빈 디렉토리)
- `.dev/qa/pending.md`     (아래 초기 템플릿 내용으로 생성)

`pending.md` 초기 템플릿:
```
---
last_updated: (날짜)
status: idle
features: []
---
```

### Step 2 — 스킬 파일 설치
다음 디렉토리를 프로젝트 루트에 생성:
- `.claude/skills/qa-master/`

이 SKILL.md 파일을 fetch해서 `.claude/skills/qa-master/SKILL.md`로 저장한다.

### Step 3 — CLAUDE.md에 추가
프로젝트 루트의 `CLAUDE.md` (없으면 생성) 에 다음 지침을 추가:

```
# QA Master
독립적으로 동작 가능한 기능 단위의 코드 변경을 완료했을 때,
"다음 작업" 또는 "계속할까요"를 묻기 전에
반드시 `.claude/skills/qa-master/SKILL.md`를 읽고 QA를 자동 수행한다.
사용자에게 QA 여부를 묻지 않는다. 실행하고 결과만 출력한다.
- 스킬 위치: .claude/skills/qa-master/SKILL.md
- 리포트 저장: .dev/qa/reports/
- 상태 파일: .dev/qa/pending.md
```

### Step 4 — 검증
- `.dev/qa/pending.md` 존재 여부 확인
- `CLAUDE.md`에 QA Master 섹션 존재 여부 확인
- 신규 설치 시 → "QA Master 셋업 완료 (v{version})" 출력
- 업데이트 시 → "QA Master 업데이트 완료: v{old} → v{new}" 출력

## 역할
기능 구현이 완성됐다고 판단되는 시점에 QA를 자동 실행하고, 모든 평가 항목이 A등급에 도달할 때까지 코드를 자동 개선하는 스킬.

## 트리거 조건
키워드 매칭이 아니라, 다음 맥락을 종합해서 CLI가 스스로 판단:
- 직전 대화에서 특정 기능 단위의 코드가 작성/수정됐는가
- 해당 코드가 독립적으로 동작 가능한 상태인가
- 사용자가 다음 작업으로 넘어가려는 흐름인가

이 세 가지가 모두 충족된다고 판단되면 **사용자에게 묻지 않고 즉시 QA를 실행**한다.

## Phase 1 — 프로젝트 분석 (자동)
- 프로젝트 유형 탐지 (RAG, CLI, API 서버, 웹앱 등)
- 핵심 기능 목록 추출 (함수/엔드포인트/모듈 단위)
- 완성된 기능 범위 특정

## Phase 2A — 기능 성능 평가
"코드가 오류 없이 의도대로 동작하는가"를 점검:
- 버그 및 런타임 오류 가능성
- 엣지케이스 처리 여부
- 예외 상황에서의 동작
- 결과: PASS / FAIL / WARN + 근거

## Phase 2B — 품질 성능 평가
"기능의 결과물이 얼마나 정확한가"를 점검.

### 평가 항목 설계 원칙
CLI가 프로젝트 유형과 기능 목적을 분석해서 평가 기준을 스스로 설계한다.
설계 시 반드시 수치 기반·측정 가능한 기준으로 작성한다:
- "좋다/나쁘다" 같은 주관적 표현 금지
- 각 항목마다 A~E 등급 기준을 수치로 정의
  예) 정답률: A=90%↑ / B=75%↑ / C=60%↑ / D=45%↑ / E=44%↓
  예) 응답 지연: A=1s↓ / B=2s↓ / C=3s↓ / D=5s↓ / E=5s↑
- 핵심 항목(Core)과 일반 항목(Standard)을 구분해서 표시

프로젝트 유형별 설계 예시:
- RAG 챗봇 → 정답률(Core), 할루시네이션 비율(Core), 응답 지연, 문서 인용 정확도
- CLI 파서 → 파싱 정확도(Core), 오류 거부율(Core), 처리 속도, 엣지케이스 커버리지
- API 서버 → 응답 정확도(Core), 상태코드 일치율(Core), 응답 지연, 페이로드 스키마 일치율

### 테스트 시나리오 생성 및 실행
1. 코드와 기능 명세를 분석해서 대표 시나리오 3~5개 직접 생성
   - 정상 케이스 (happy path)
   - 경계값 케이스
   - 의도적 오입력 케이스
2. 각 시나리오를 실행하고 결과를 수집
3. 수집된 결과를 설계한 등급 기준에 대입해서 항목별 등급 산출
4. 핵심 항목과 일반 항목의 등급을 종합해서 최종 등급(A~E) 결정

### dev/prod PASS 기준
환경변수 `QA_MODE=dev|prod` 로 결정. 기본값은 `dev`.

| 모드 | 일반 항목 | 핵심(Core) 항목 |
|------|-----------|----------------|
| dev  | 전 항목 C등급 이상 | 핵심 항목 B등급 이상 |
| prod | 전 항목 B등급 이상 | 핵심 항목 A등급 이상 |

기준 미달 항목이 하나라도 있으면 Phase 2B는 FAIL로 처리한다.

## Phase 2C — 자동 개선 루프 (신규)
Phase 2A + 2B 완료 후, A등급 미달 항목이 하나라도 있으면 자동으로 개선 루프에 진입한다.

### 루프 흐름
```
라운드 1: Phase 2A + 2B 실행 → 전체 항목 확인
  → 모든 항목 A등급? → YES → PASS → Phase 3으로 이동
  → NO → A등급 미달 항목 식별 → 원인 분석 → 코드 자동 수정 → Phase 2A + 2B 재실행

라운드 2: 재평가
  → 모든 항목 A등급? → YES → PASS → Phase 3으로 이동
  → NO → 다시 자동 수정 → 재실행

라운드 3: 재평가
  → 모든 항목 A등급? → YES → PASS → Phase 3으로 이동
  → NO → 루프 중단 → 현재 상태로 Phase 3 진행
```

### 루프 규칙
1. **목표**: 모든 평가 항목(Core + Standard)이 A등급에 도달
2. **최대 라운드**: 3회 (최초 평가가 라운드 1)
3. **라운드별 수행 내용**:
   - A등급 미달 항목 전체 식별
   - 각 항목의 근본 원인 분석
   - 타겟 코드 수정 적용
   - Phase 2A + 2B 재실행으로 개선 효과 측정
4. **루프 종료 조건**:
   - 모든 항목 A등급 도달 → PASS로 종료
   - 라운드 3 완료 후에도 A등급 미달 → 현재 상태로 종료
5. **최대 라운드 초과 시**:
   - 가장 좋은 결과를 리포트에 저장
   - 남은 이슈 요약을 사용자에게 출력
   - 이후 대응은 사용자가 수동으로 결정

### 개선 전략
코드를 수정해서 등급을 올릴 때:
- 한 번에 하나의 근본 원인만 수정 — 산탄식 변경 금지
- Core 항목을 Standard 항목보다 우선 수정
- 기존 통과 항목의 동작을 유지 — 퇴행(regression) 금지
- 각 수정은 최소 범위로, 타겟을 명확히

## Phase 3 — 리포트 저장
- `.dev/qa/reports/{feature-name}.md` 로 저장
- 2A, 2B, 2C(루프 이력) 결과를 각각 섹션으로 구분해서 기록
- `.dev/qa/pending.md` 상태 업데이트
- 기능이 수정/삭제되면 해당 리포트도 최신화/삭제

### 리포트 수명 주기 관리
QA 세션 시작 시, CLI는 기존 리포트를 현재 코드베이스와 대조해서 확인:
- 리포트된 기능이 코드베이스에 더 이상 존재하지 않으면 → 리포트 삭제
- 리포트된 기능이 크게 수정됐으면 → QA를 재실행하고 리포트 덮어쓰기
- 리포트가 없는 새 기능이 감지되면 → `pending.md`에 QA 후보로 추가

"크게 수정됐다"는 기준: 핵심 로직, 입출력 인터페이스, 동작이 변경된 경우. 단순 리팩토링이나 포매팅 변경은 해당 없음.
